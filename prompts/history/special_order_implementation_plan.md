# Special Order System — Implementation Plan

## Context

The game has a stub special-order infrastructure on `MerchantData`
(`special_order_pool`, `special_order_count`, `special_order_bonus`) and a
daily rolling function in `MerchantRegistry.roll_special_orders()`. None of it
is wired into the UI or sale flow. This plan replaces that scaffolding with a
full panel-based order system driven by designer-authored `SpecialOrderData`
templates: each merchant holds a pool of templates (e.g. one premium, one
bulk), rolls new orders on its own cadence, and the player fulfills them in a
dedicated panel with save/load persistence.

## 1. New Data Classes

### `data/definitions/special_order_data.gd`

Designer-authored Resource template. Place `.tres` files under
`data/tres/special_orders/`. Naming convention: `{merchant_id}_{archetype}.tres`
(e.g. `antique_dealer_premium.tres`, `antique_dealer_bulk.tres`).

```gdscript
class_name SpecialOrderData
extends Resource

# Internal identifier. snake_case. Matches the .tres filename stem.
@export var special_order_id: String = ""

# ── Slot generation ──────────────────────────────────────────────────────────

@export var slot_count_min: int = 1
@export var slot_count_max: int = 1
@export var required_count_min: int = 1
@export var required_count_max: int = 1

# Pool of categories a generated slot may target.
@export var allowed_categories: Array[CategoryData] = []

# Probability that a generated slot gets a rarity gate (>= UNCOMMON).
@export var rarity_gate_chance: float = 0.0

# Probability that a generated slot gets a condition gate (>= 0.6).
@export var condition_gate_chance: float = 0.0

# ── Pricing & turn-in flags ──────────────────────────────────────────────────

@export var buff_min: float = 1.0
@export var buff_max: float = 1.0

# If true, per-item price uses entry.get_condition_multiplier().
# Otherwise condition is ignored (flat-rate bulk).
@export var uses_condition_pricing: bool = false

# If true, confirm works with any non-empty assignment and slot progress
# persists across sessions. If false, confirm is disabled until every slot
# is fully filled in one session.
@export var allow_partial_delivery: bool = false

# ── Completion ───────────────────────────────────────────────────────────────

@export var completion_bonus: int = 0
@export var deadline_days: int = 5
```

### `game/shared/special_order/order_slot.gd`

Runtime-only class. Generated from a `SpecialOrderData` template, persisted via
`to_dict()` / `from_dict()`.

```gdscript
class_name OrderSlot
extends RefCounted

var category: CategoryData
var min_rarity: int = -1          # -1 = no gate; otherwise ItemData.Rarity value
var min_condition: float = 0.0    # 0.0 = no gate
var required_count: int = 1
var filled_count: int = 0

func remaining() -> int
func is_full() -> bool
func accepts(entry: ItemEntry) -> bool
    # entry.item_data.category_data == category (same registry instance ref compare)
    # entry.item_data.rarity >= min_rarity (when min_rarity >= 0)
    # entry.condition >= min_condition (when min_condition > 0)

func to_dict() -> Dictionary
    # {"category_id": category.category_id, "min_rarity": ..., "min_condition": ...,
    #  "required_count": ..., "filled_count": ...}

static func from_dict(d: Dictionary) -> OrderSlot
    # Resolves category_id back to CategoryData via a category registry.
    # (If none exists, use ItemRegistry or add a lightweight CategoryRegistry
    # helper. CategoryData is already loaded transitively via ItemData.)
```

### `game/shared/special_order/special_order.gd`

Runtime-only class. Generated by `MerchantRegistry._generate_order()` from a
`SpecialOrderData` template.

```gdscript
class_name SpecialOrder
extends RefCounted

var id: String                    # "{merchant_id}_{counter}"
var special_order_id: String      # source SpecialOrderData.special_order_id
var merchant_id: String
var slots: Array[OrderSlot]
var buff: float
var completion_bonus: int
var deadline_day: int             # absolute day
var uses_condition_pricing: bool
var allow_partial_delivery: bool

func is_complete() -> bool
    # all slots full

func is_expired(current_day: int) -> bool
    # current_day > deadline_day

func compute_item_price(entry: ItemEntry) -> int
    # uses_condition_pricing:
    #   base_value * entry.get_condition_multiplier() * buff
    # else:
    #   base_value * buff
    # Market factor is ignored in both cases.

func to_dict() -> Dictionary
static func from_dict(d: Dictionary) -> SpecialOrder
```

## 3. MerchantRegistry Changes

**File:** `global/autoload/merchant_registry.gd`

**Replace** `roll_special_orders()` with `_advance_orders()`:

```gdscript
func advance_day() -> void:
    _advance_orders()
    _reset_negotiations()


func _advance_orders() -> void:
    var day: int = SaveManager.current_day
    for m: MerchantData in _merchants.values():
        # 1. Clear expired orders (no payout)
        var kept: Array[SpecialOrder] = []
        for order: SpecialOrder in m.active_orders:
            if not order.is_expired(day):
                kept.append(order)
        m.active_orders = kept

        # 2. Roll new order if eligible
        if m.order_roll_cadence <= 0:
            continue
        if m.special_orders.is_empty():
            continue
        if m.last_order_roll_day >= 0 and (day - m.last_order_roll_day) < m.order_roll_cadence:
            continue
        if m.active_orders.size() >= m.max_active_orders:
            continue

        var order := _generate_order(m, day)
        if order != null:
            m.active_orders.append(order)
            m.last_order_roll_day = day
```

**Add** `_generate_order(m, day) -> SpecialOrder`:

```
1. template = m.special_orders.pick_random()
2. If template.allowed_categories is empty, return null.
3. slot_count = randi_range(template.slot_count_min, template.slot_count_max)
4. For each slot:
   - category = template.allowed_categories.pick_random()
   - required_count = randi_range(template.required_count_min, template.required_count_max)
   - min_rarity = ItemData.Rarity.UNCOMMON if randf() < template.rarity_gate_chance else -1
   - min_condition = 0.6 if randf() < template.condition_gate_chance else 0.0
5. buff = randf_range(template.buff_min, template.buff_max)
6. order_id = "%s_%d" % [m.merchant_id, _next_order_id]; _next_order_id += 1
7. special_order_id = template.special_order_id
8. deadline_day = day + template.deadline_days
9. Copy uses_condition_pricing, allow_partial_delivery, completion_bonus from template.
10. Return the assembled SpecialOrder.
```

**Add** query helpers:

```gdscript
func get_active_orders(merchant: MerchantData) -> Array[SpecialOrder]
func has_active_orders(merchant: MerchantData) -> bool
```

**Add** `_next_order_id: int = 0` counter (persisted via SaveManager).

### Scene Tree

```
FulfillmentPanel (Control)
└── RootVBox (VBoxContainer)
    ├── TitleLabel (Label)
    ├── ContentHBox (HBoxContainer)
    │   ├── LeftVBox (VBoxContainer)         — order list + details
    │   │   ├── OrderListVBox (VBoxContainer) — one button per active order
    │   │   ├── OrderInfoVBox (VBoxContainer) — buff, bonus, deadline, flags
    │   │   └── SlotsVBox (VBoxContainer)     — one row per slot (clickable)
    │   └── RightVBox (VBoxContainer)         — inventory + summary
    │       ├── ItemListPanel (ItemListPanel) — filtered inventory
    │       └── SummaryVBox (VBoxContainer)
    │           ├── SessionPayoutLabel (Label)
    │           └── CompletionBonusLabel (Label)
    └── Footer (HBoxContainer)
        ├── ConfirmButton (Button)
        └── BackButton (Button)
```

### State

```gdscript
var _merchant: MerchantData
var _selected_order: SpecialOrder           # currently viewed order
var _selected_slot_index: int = -1          # active slot for assignment
var _session_assignments: Dictionary        # slot_index -> Array[ItemEntry]
```

### UI Flow

1. `_ready()`: consume merchant, populate order list buttons.
2. Clicking an order button → `_select_order(order)` → show info + slots.
3. Clicking a slot → `_select_slot(index)` → filter inventory via
   `slot.accepts()`, populate `ItemListPanel`.
4. Clicking an item row → assign / unassign to the current slot (toggle).
5. Assignment capped at `slot.remaining()` per slot.
6. Session payout label updates live: sum of `order.compute_item_price(entry)`
   for all assigned items.
7. Completion bonus label: shown when this session would complete the order.

### Confirm Logic

The confirm flow branches on `order.allow_partial_delivery`, not on a hardcoded
archetype:

**`allow_partial_delivery == true`:**

- Enabled when any items are assigned this session.
- On confirm:
  - Payout = sum of per-item prices for assigned items.
  - For each slot, `filled_count += session_assigned_count`.
  - Remove assigned items from `SaveManager.storage_items`.
  - Add knowledge points per item (reuse normal-sale hook).
  - `SaveManager.cash += payout`.
  - If order now complete: credit `completion_bonus`, append `order_id` to
    `merchant.completed_order_ids`, remove from `active_orders`.
  - Save and return to merchant hub.

**`allow_partial_delivery == false`:**

- Enabled only when, for every slot,
  `filled_count + session_assigned_count == required_count`.
- On confirm:
  - Payout = sum of per-item prices + `completion_bonus`.
  - Remove all assigned items from storage.
  - Add knowledge points per item.
  - `SaveManager.cash += payout`.
  - Append `order_id` to `merchant.completed_order_ids`, remove from
    `active_orders`.
  - Save and return to merchant hub.

## 12. Verification

- Advancing a day past a merchant's roll cadence generates a new order; it
  appears in `active_orders` with deadline and buff visible.
- Merchant hub "Orders (N)" button enables when `N > 0` and is hidden when
  `order_roll_cadence == 0`.
- Fulfillment panel shows order info (buff, bonus, deadline, slot list).
- Selecting a slot filters inventory through `slot.accepts()`; wrong-category,
  low-rarity, or low-condition items do not appear.
- Assigning items updates the session payout label live.
- `allow_partial_delivery == false` (Premium): confirm disabled until all
  slots fully filled; on confirm, payout = item sum + completion bonus.
- `allow_partial_delivery == true` (Bulk): confirm enabled with any
  assignment; partial slot progress persists to the next panel open.
- An expired order disappears on the next `advance_day()` with no payout,
  even if a bulk order had partial progress.
- Save mid-order → reload → `slots` (including `filled_count`), `deadline_day`,
  `buff`, `completion_bonus`, `uses_condition_pricing`, and
  `allow_partial_delivery` all unchanged.
- Completed orders accumulate in `merchant.completed_order_ids` and persist
  across saves.

## 13. Edge Cases

1. **Bulk `filled_count` and consumed items**: `filled_count` only increments
   on confirm, at which point items are also removed from storage. No
   validation needed on panel open — consumed items are already gone.
2. **No cross-order item conflict**: session assignments are local (not
   persisted). Items are consumed only on confirm. Only one panel is open at a
   time, so no item can be double-assigned.
3. **Empty `allowed_categories` on a template**: `_generate_order()` returns
   null — no order generated. Legal at authoring time.
4. **Empty `special_orders` on a merchant**: `_advance_orders()` skips rolling.
   This is the pawn-shop path.
5. **Save migration**: old saves lacking `merchant_orders` / `next_order_id`
   default to empty / `0` via `parsed.has()` guards. Old `special_order_*`
   fields were on `.tres` resources, not in the save file — no conflict.
6. **Multi-day advance**: `advance_day()` is called once regardless of days
   elapsed. The cadence check `(day - last_roll_day) >= cadence` handles gaps
   correctly — at most one order rolls per call.
7. **Bulk partial delivery persists**: confirm increments `filled_count`,
   removes consumed items, keeps the order active until fully complete or
   expired.
8. **Premium all-or-nothing**: session-only assignments. Leaving without
   confirming loses no persistent state (no `filled_count` changes until
   confirm).
9. **Category identity on reload**: `OrderSlot.from_dict()` resolves saved
   `category_id` back to the registry `CategoryData` instance. Assignment
   accept checks compare by reference — this works as long as the registry
   returns the same loaded instance (which it does for resources loaded via
   `ResourceDirLoader`).
